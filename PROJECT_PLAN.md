# CV → MES Agentic Defect Detection — Task Plan

Two-person build. **A = you** (agentic side) · **B = coworker** (DB + plumbing side).
The system: steel-defect camera (YOLO, steelMLOps repo) → bridge → Claude agent (this repo) → live dashboard.
Tasks are in execution order. Each has a *why* — the how gets figured out (or taught) when you get there.

## Stage 1 — Foundation (do these together, ~1 day)

**1. Train the YOLO model on Colab — B**
The repo ships no trained model file (`best.pt`), so the `/predict` endpoint currently has no brain. Nothing else in the project can even be tested until this exists. ~20 min on Colab's free GPU via `notebooks/steel_defect_colab.ipynb`; place the file at `runs/detect/steel_defect_colab_50_epochs/weights/best.pt`.

**2. Verify the inference API with one test image — B**
Start `deployment/api.py`, POST an image from `data/dataset/images/test/`, confirm you get detections back. Why: proves the model, its dependencies, and the service all work *before* you build five things on top of them. If something's broken, you find out now, with one moving part, not later with five.

**3. Write CONTRACTS.md — A drafts, B approves**
One page: the two table layouts, the shape of the detection payload, the signature of `analyze_batch(detections)`, and which port each program uses (8080 API / 8081 bridge / 8501 dashboard). Why: this is what lets two people work in parallel without breaking each other. Almost every two-person integration failure is a disagreement about a column name or payload field that a 30-minute contract would have prevented. A must approve because `analyze_batch()` defines A's input.

## Stage 2 — Plumbing before brain (B builds, A learns — in parallel)

**4. Create the two database tables — B**
`VisionDetections` (every camera sighting) and `AgentAlerts` (every investigation and its status/report). Why: this is the system's shared memory — the bridge writes to it, the agent reads and writes to it, the dashboard displays it. Must be created with "IF NOT EXISTS" at startup because `mes.db` gets wiped and regenerated by the data generator.

**5. Lookup helpers — B**
`get_frame_machines()` and `get_active_work_order(machine_id)`, querying the live database. Why: a detection saying "scratch, 87%" is useless on its own — it becomes investigable only when tied to a machine and work order, because those are the keys the agent uses to correlate with maintenance history, QC records, and material lots. Querying live (not hardcoding IDs) keeps the links valid after every DB regeneration.

**6. Read the existing agent code — A** *(parallel with 4–5)*
Read `app_factory/mes_agents/mes_analysis_agent.py` and `tools/database_tools.py` end to end. Why: in Stage 3 you design your own tool and prompts; the three existing tools are working examples of exactly the pattern you'll follow. One hour of reading here replaces days of confusion later. This is the start of your agentic-design education, on real, working code.

**7. Fake camera (replay script) — B**
Sends a NEU test image to `/predict` every ~5s, forwards results + a machine assignment to the bridge; includes a burst mode (5–8 same-defect images inside a minute). Why: there is no real camera — the simulator is the system's heartbeat, and making it *controllable* (bursts on demand) is what lets you test batching (task 8) and demo convincingly (task 14). Burst mode later aligns with generator-injected incidents for evaluation (task 15).

**8. Bridge: confidence gate + 30-second batching — B**
FastAPI service: save every detection, keep only confidence ≥ 0.80, collect a burst for 30s, emit one batch. Why the gate: low-confidence detections are mostly false alarms; every batch triggers a ~60-second, real-money Claude run, so junk in = cost and noise out. Why batching: a defect cluster produces 20 detections in a minute — that is *one* event needing *one* investigation, not 20 parallel ones. This single design decision is the difference between a usable system and an expensive spammy one.

## Stage 3 — The brain (A builds; B available for DB questions)

**9. Wire the seam with a stub — both, 30 minutes**
B's bridge calls a dummy `analyze_batch()` that just prints the batch. Why: proves the connection between your two halves works *before* the LLM enters. From here on, any bug is clearly on one side of the seam or the other — this cheap checkpoint saves hours of two-person debugging.

**10. Agent runner — A**
Real `analyze_batch()`: turn the batch into an investigation prompt ("6 scratches, Machine 12, WorkOrder 4471, 14:02–14:03 — correlate with QC records, downtime, material lots; isolated or pattern? root cause?"), invoke the existing Strands agent, write the report into `AgentAlerts`. Why: this is the moment the project becomes *agentic* — detection stops being a log entry and starts being an investigation.

**11. Design and implement your tool — A**
E.g. `get_recent_detections(machine_id, hours)`, registered with the agent. Why: without it the agent only sees the current batch and can never answer "is this recurring?" — the single most valuable question. A tool is how you extend what an agent *can find out*, and designing one (what it does, what its description tells Claude, when Claude reaches for it) is the core skill you set out to learn.

**12. Prompt iteration — A (ongoing)**
Run real batches, read the reports, notice what's generic or wrong, refine the prompt, repeat. Why: first-draft prompts produce plausible-sounding but shallow reports. Quality in agentic systems comes from this loop, not from the first version — and doing it builds the judgment that separates "called an API" from "designed an agent."

**13. Guardrails — A**
One agent run at a time, a max-runs-per-hour cap, failures mark the alert `failed` instead of crashing. Why: each run costs real money and takes a minute. A burst storm with no guardrails means surprise bills and pile-ups; a crash without failure-handling silently kills the whole pipeline.

## Stage 4 — Visible and trustworthy

**14. Live dashboard tab — A**
Auto-refreshing tab: alert appears *instantly* as ⚠ pending, agent's report fills in ~a minute later, detection feed below. Why: the system is invisible without it, and the instant-alarm / explanation-later transition is the entire demo — it's the visual proof that the LLM is not in the alert path. Built last because it only *reads* tables that must exist and fill first.

**15. Evaluation against ground truth — A**
The data generator secretly injects quality incidents (specific machine + defect category). Align simulator bursts with them, run ~10 scenarios, score the agent: right machine? right correlated cause? any hallucination? Record results in `docs/evaluation.md`. Why: this is the only way to know the reports are *correct* rather than merely fluent — and "measured agent accuracy against known ground truth" is the claim that makes this project credible to an interviewer, because almost no LLM side-project can make it.

**16. One-command startup script — B**
Launches API, bridge, dashboard, simulator together. Why: four terminals is where live demos go to die. One command means anyone — including the interviewer you send the repo to — can run it.

**17. README + demo GIF — A**
Architecture diagram, the GIF of a burst becoming an alert becoming a report, latency numbers (detect ~80ms / alert <1s / analysis ~60s), honest limitations. Why: it's the only part most people will ever see. A working system with no README reads as a tutorial; a clear README with a GIF and an evaluation table reads as engineering.
